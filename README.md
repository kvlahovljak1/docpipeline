# DocPipeline

A document processing system for invoices and purchase orders. Extracts structured data, validates it, and stores it in a hosted database — built entirely without a paid AI API.

---

## Quick Start

### 1. Create the database table in Supabase

1. Go to [supabase.com](https://supabase.com) and sign in
2. Open your project → **SQL Editor** → **New query**
3. Paste the contents of `supabase/schema.sql` and click **Run**

### 2. Install dependencies

```bash
npm install
```

This also automatically copies `pdf.worker.min.js` into `public/` (required for PDF support).

### 3. Start

```bash
npm start
```

Opens at [http://localhost:3000](http://localhost:3000)

## What It Does

Upload a document → the system extracts fields → validates the data → flags issues → you review and confirm or reject.

### Supported formats

| Format | Method |
|---|---|
| `.txt` | Regex pattern matching on free text |
| `.csv` | Column/header detection + key-value parsing |
| `.pdf` | pdf.js text layer extraction |
| `.png` / `.jpg` | Tesseract.js OCR (~10s per image) |

### Fields extracted

- Document type (invoice or purchase order)
- Supplier / company name
- Document number
- Issue date and due date
- Currency
- Line items (description, qty, unit price, total)
- Subtotal, tax, total

### Validation checks

Every document is run through a pure JavaScript validation engine (no AI involved):

- Missing required fields (supplier, document number, issue date, currency, total)
- Due date is before issue date
- Suspicious year (outside 2000–2099)
- Line item math — `qty × unit price` must equal stated line total
- Subtotal cross-check — sum of all line items must equal stated subtotal
- Total cross-check — `subtotal + tax` must equal stated total
- Duplicate document numbers across all uploaded files

Documents that pass all checks are marked **Validated**. Documents with issues are marked **Needs Review** and the specific issues are shown in the review panel.

### Status workflow

```
Uploaded → Needs Review → Validated
                       ↘ Rejected
```

## Architecture Decisions

### Why regex extraction instead of an AI API?

The obvious approach for extracting structured data from documents is to send them to an LLM (GPT-4, Claude, etc.) and get back JSON. It's more accurate, handles messy/unusual formats better, and requires almost no code.

We didn't do this for two reasons:

1. **Cost** — Every document upload would cost money (~$0.003 per doc with Claude Sonnet). For a portfolio project or small business tool, that adds up and requires a credit card and API account just to run the app.

2. **Simplicity** — Regex extraction runs entirely in the browser with zero network requests, zero latency beyond the file read itself, and no dependency on an external service being up.

The tradeoff is accuracy. Regex works well on structured and semi-structured documents (anything generated by software — accounting tools, Excel exports, PDF generators). It fails on genuinely messy inputs: handwritten notes, scanned documents with poor OCR, or non-standard layouts.

**What we'd do with a budget:** Send the extracted text to Claude with a structured extraction prompt. The extraction quality jumps dramatically, especially for edge cases. Cost would be negligible at low volume.

### Why Tesseract.js instead of a cloud OCR API?

Tesseract.js is an open-source OCR engine that runs entirely in the browser via WebAssembly. No API key, no cost, no backend.

The downsides are significant:
- **Slow** — ~5–15 seconds per image depending on size and device
- **Inaccurate on complex layouts** — tables, coloured backgrounds, logos, multi-column layouts all confuse it
- **Large** — adds ~10MB to the initial load

Cloud OCR alternatives (Google Cloud Vision, AWS Textract, Azure Document Intelligence) are dramatically better. Textract in particular is purpose-built for invoices and purchase orders — it understands table structure natively and returns key-value pairs and line items already parsed. Cost is roughly $0.001–$0.015 per page depending on the service and features used.

**What we'd do with a budget:** Replace Tesseract with AWS Textract or Google Document AI via a small backend proxy. Accuracy on real-world invoice images would go from ~60% to ~95%+.

### Why Supabase instead of a local database?

SQLite running locally would have been simpler to set up (no account, no network, just a file). We chose Supabase because:

1. The data persists across devices and browsers — not just the machine running the app
2. It's a real hosted PostgreSQL instance, which is closer to a production setup
3. The free tier is genuinely free — 500MB storage, no credit card required

The tradeoff is that the app won't work offline and requires the Supabase project to exist before first run.

**What we'd do in production:** Keep Supabase but add row-level security policies tied to authenticated users (right now the table allows all reads/writes from anyone with the anon key, which is fine for a single-user local tool but not for a multi-user deployment).

### Why no backend server?

The app calls Supabase directly from the browser using the anon/public key. This works because Supabase's Row Level Security is designed for exactly this — the anon key is safe to expose as long as your RLS policies are correct.

For the Anthropic API (if we were using it), you'd need a backend to keep the key secret. We avoided this by not using a paid API.

**What we'd do in production:** Add a lightweight Express or Next.js API layer. This would let us: keep all API keys server-side, add authentication, run heavier processing (e.g. multi-page PDF parsing), and add a job queue for batch uploads.

### Why pdf.js instead of a server-side PDF parser?

pdf.js runs in the browser — no server needed, no file uploads to an external service. It extracts the text layer from PDFs generated by software (Word, accounting tools, etc.).

The limitation is scanned PDFs. A scanned invoice is just an image inside a PDF container — there's no text layer to extract. pdf.js returns nothing useful. For these, you'd need to rasterise each page and run OCR, which is what cloud services like AWS Textract do automatically.

**What we'd do with a budget:** Detect when a PDF has no text layer and automatically fall back to page-level OCR using a cloud API.

## What Could Be Better

| Area | Current | Better approach |
|---|---|---|
| Extraction accuracy | Regex | Claude/GPT-4 API with structured output |
| Image OCR | Tesseract.js (slow, ~60% accurate) | AWS Textract or Google Document AI |
| Scanned PDFs | Not supported | Page rasterisation + cloud OCR |
| Auth | None — single user only | Supabase Auth with email/password |
| Multi-user | Not supported | RLS policies per user ID |
| File storage | Not stored — only extracted text | Supabase Storage for original files |
| Batch processing | Sequential, one at a time | Job queue (e.g. BullMQ) |
| Export | None | CSV/Excel export of extracted data |
| Audit trail | None | Log every status change with timestamp |
| Tests | None | Jest unit tests for the validation engine |
| Deployment | Local only | Vercel (frontend) + Supabase (database) |

## Project Structure

```
src/
  lib/
    extractor.js    ← file reading + data extraction (txt, csv, pdf, image)
    validation.js   ← pure JS validation logic
    supabase.js     ← database read/write operations
  components/
    App.jsx         ← root component, state management
    Dashboard.jsx   ← stats cards + document table
    ReviewPanel.jsx ← edit modal with issue display
    UploadZone.jsx  ← drag-and-drop file input
    StatusBadge.jsx ← status pill component
supabase/
  schema.sql        ← run this in Supabase SQL editor on first setup
scripts/
  copy-pdfjs-worker.js  ← postinstall: copies pdf.js worker to public/
```
